## 🎯 第一阶段：理解 RAG 原理

### 1.1 什么是 RAG？

#### 通俗解释：闭卷 vs 开卷考试

**传统 LLM（闭卷考试）**：
```
你：北京大学的图书馆开放时间是？
GPT：我不知道，我的训练数据截止到 2023 年...
```

**RAG（开卷考试）**：
```
你：北京大学的图书馆开放时间是？

系统内部：
1. 在知识库中搜索"北京大学 图书馆 开放时间"
2. 找到相关文档："图书馆周一至周五 8:00-22:00 开放"
3. 把文档和问题一起给 GPT

GPT：根据资料，北京大学图书馆周一至周五 8:00-22:00 开放。
```

#### RAG 的三个组成部分

**RAG = Retrieval（检索）+ Augmented（增强）+ Generation（生成）**

1. **Retrieval（检索）**：从知识库中找到相关文档
2. **Augmented（增强）**：把检索到的文档作为上下文
3. **Generation（生成）**：LLM 基于上下文生成回答

#### 为什么需要 RAG？

| 问题 | 传统 LLM | RAG 方案 |
|------|---------|---------|
| 知识过时 | ❌ 训练数据有截止日期 | ✅ 实时更新知识库 |
| 专业领域 | ❌ 通用知识，不够专业 | ✅ 可添加专业文档 |
| 幻觉问题 | ❌ 容易编造不存在的信息 | ✅ 基于真实文档回答 |
| 成本 | ❌ 需要重新训练模型 | ✅ 只需更新文档 |

### 1.2 RAG 完整工作流程

#### 构建阶段（离线）

```
原始文档（data/*.md）
    ↓
【步骤1】文档加载器读取
    ↓
【步骤2】切分成小块（chunk）
    ↓
【步骤3】转换成向量（embedding）
    ↓
【步骤4】存入向量数据库（FAISS）
    ↓
保存到磁盘（storage/faiss_index）
```

**对应代码**：`scripts/build_index.py`

#### 问答阶段（在线）

```
用户提问："图书馆开放时间？"
    ↓
【步骤1】问题转成向量
    ↓
【步骤2】在向量库中找相似文档（top-k）
    ↓
【步骤3】拼接 Prompt
    """
    上下文：[检索到的文档片段]
    问题：图书馆开放时间？
    请回答：
    """
    ↓
【步骤4】发送给 LLM
    ↓
【步骤5】LLM 生成回答
    ↓
返回给用户
```

**对应代码**：`app/rag.py` + `app/main.py`

### 1.3 核心概念速查

#### Chunk（文档切分）

**定义**：把长文档切成小块，每块包含一个完整的语义单元。

**为什么要切分？**
- ❌ 直接全部给 LLM → Token 太多，成本高，效果差
- ✅ 切成小块 → 只检索相关的几块，精准且便宜

**关键参数**：
```python
chunk_size=500      # 每块最多 500 个字符
chunk_overlap=80    # 块之间重叠 80 个字符（避免切断句子）
```

#### Embedding（向量化）

**定义**：把文字转成数字向量，让计算机能"理解"语义。

**示例**：
```
"图书馆开放时间" → [0.2, 0.8, 0.1, ..., 0.5]（1536个数字）
"图书馆营业时间" → [0.21, 0.79, 0.11, ..., 0.49]（很接近！）
"今天天气真好"   → [0.9, 0.1, 0.7, ..., 0.3]（完全不同）
```

**关键特性**：相似的文字 → 相似的向量

#### Vector Store（向量数据库）

**定义**：存储和快速搜索向量的数据库。

**本项目使用**：FAISS（Facebook AI Similarity Search）

**功能**：
```
输入：问题向量 [0.2, 0.8, ...]
输出：最相似的 k 个文档片段
```

#### Retriever（检索器）

**定义**：从向量数据库中检索相关文档的组件。

**代码示例**：
```python
retriever = vectorstore.as_retriever(search_kwargs={"k": 4})
docs = retriever.invoke("图书馆开放时间？")
# 返回最相关的 4 个文档块
```

### 2.2 文档切分（RecursiveCharacterTextSplitter）

#### 为什么要切分？

假设有一个 10 页的文档：
- ❌ 直接全部给 GPT → 10000 tokens，成本 $0.01，且可能超出上下文限制
- ✅ 切成 20 块，只检索相关的 4 块 → 2000 tokens，成本 $0.002

#### 代码解析：build_index.py 第29-34行

```python
splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,      # 每块最多 500 个字符
    chunk_overlap=80,    # 相邻块重叠 80 个字符
    separators=["\n\n", "\n", "。", "，", " ", ""],  # 优先在这些位置切分
)
chunks = splitter.split_documents(docs)
```

#### separators 的作用

`RecursiveCharacterTextSplitter` 会按优先级尝试切分：

1. 先尝试在 `\n\n`（段落）处切
2. 不行就在 `\n`（换行）处切
3. 再不行就在 `。`（句号）处切
4. 再不行就在 `，`（逗号）处切
5. 最后才在字符中间切

**好的切分示例**：
```
块1: "图书馆周一至周五8:00-22:00开放。"
块2: "周末9:00-18:00开放。寒暑假时间另行通知。"
```

**坏的切分示例**：
```
块1: "图书馆周一至周五8:00-22:00开"
块2: "放。周末9:00-18:00开放。寒暑假"
```

#### chunk_overlap 的作用

**没有 overlap**：
```
块1: "...选课时间为开学第一周。"
块2: "补考安排在第3-5周..."
```
如果用户问"选课后多久补考"，可能检索不到完整信息。

**有 overlap（80字符）**：
```
块1: "...选课时间为开学第一周。补考安排在第3-5周..."
块2: "补考安排在第3-5周。具体时间..."
```
两个块都包含关键信息，检索更准确。

## 🧠 第三阶段：向量化原理

### 3.1 什么是向量（Embedding）？

#### 文字 → 数字的转换

计算机无法直接理解文字，需要转换成数字：

```
"图书馆开放时间" → [0.234, 0.891, 0.123, ..., 0.456]
                    ↑
                    1536个浮点数
```

#### 1536维向量的含义

- 每个维度代表文本的某个"特征"
- 这些特征是 AI 模型自动学习的（不是人工定义）
- 维度越高，表达能力越强

**类比**：
- 1维：只能表示"长度"
- 2维：可以表示"长度+宽度"
- 1536维：可以表示文本的语义、情感、主题等复杂特征

#### 为什么相似的文字有相似的向量？

OpenAI 的 embedding 模型经过大量训练，学会了：
- "图书馆开放时间" 和 "图书馆营业时间" 是同一个意思
- "补考" 和 "重修" 是相关概念
- "天气" 和 "图书馆" 完全无关

**实验证明**：
```
"图书馆开放时间" vs "图书馆营业时间" → 相似度 0.817（很高）
"图书馆开放时间" vs "今天天气真好"   → 相似度 0.222（很低）
```

### 3.3 相似度计算

#### 余弦相似度公式

```python
def cosine_similarity(vec1, vec2):
    """计算两个向量的余弦相似度"""
    return np.dot(vec1, vec2) / (np.linalg.norm(vec1) * np.linalg.norm(vec2))
```

**结果范围**：0 到 1
- 1.0 = 完全相同
- 0.8+ = 非常相似
- 0.5-0.8 = 有一定相关性
- 0.3以下 = 基本不相关

#### 实验结果：相似度矩阵

```
                                        文本1  文本2  文本3  文本4
文本1: 图书馆周一到周五开放时间...        1.000  0.817  0.380  0.222
文本2: 图书馆的营业时间是什么            0.817  1.000  0.291  0.162
文本3: 补考安排在每学期开学后...         0.380  0.291  1.000  0.102
文本4: 今天天气真好                     0.222  0.162  0.102  1.000
```

#### 关键发现

1. **语义相似性**
   - 文本1 和 文本2：相似度 0.817
   - 虽然用词不同（"开放时间" vs "营业时间"），但语义相同
   - AI 理解了它们在问同一件事

2. **主题相关性**
   - 文本1 和 文本3：相似度 0.380
   - 都是校园相关，但主题不同（图书馆 vs 补考）
   - 有一定相关性，但不高

3. **完全无关**
   - 文本4（天气）和其他文本：相似度 0.1-0.2
   - 完全不同的主题
   - 这就是为什么检索时不会返回无关内容

💡 **这就是 RAG 检索的核心原理**：找到向量最相似的文档！

## 🔍 第四阶段：向量检索

### 4.1 FAISS 向量数据库

#### 什么是 FAISS？

**FAISS**（Facebook AI Similarity Search）是 Meta 开源的向量检索库。

**特点**：
- 快速：可以在百万级向量中毫秒级检索
- 高效：内存占用小
- 灵活：支持多种索引类型

#### 生成的文件

运行 `build_index.py` 后，在 `storage/faiss_index/` 目录下生成：

```
index.faiss (43KB)  - FAISS 向量索引（二进制文件）
index.pkl (7.1KB)   - 文档内容和元数据（Python pickle 格式）
```

**为什么要保存到磁盘？**
- 避免每次启动都调用 OpenAI API（省钱）
- 加载速度快（几毫秒）
- 可以增量更新

#### 如何加载已有索引

```python
from langchain_community.vectorstores import FAISS
from langchain_openai import OpenAIEmbeddings

embeddings = OpenAIEmbeddings(...)
vectorstore = FAISS.load_local(
    "storage/faiss_index",
    embeddings,
    allow_dangerous_deserialization=True
)
```

⚠️ **注意**：`allow_dangerous_deserialization=True` 是因为 pickle 文件可能包含恶意代码，只加载可信来源的索引。

### 4.4 检索的局限性

#### 向量检索的优势

✅ 语义理解：能理解"开门"="开放时间"="营业时间"
✅ 模糊匹配：不需要精确的关键词
✅ 跨语言：理论上支持多语言检索

#### 向量检索的局限

❌ 无法判断"没有答案"：总是返回 k 个结果
❌ 可能检索到不相关内容：相似度低时仍会返回
❌ 无法理解复杂逻辑：如"除了...之外"

#### 为什么需要 LLM？

LLM 的作用：
1. **判断相关性**：检索到的文档是否真的回答了问题
2. **综合信息**：从多个文档块中提取关键信息
3. **生成回答**：用自然语言组织答案
4. **处理边界情况**：知识库中没有答案时，礼貌地告知用户

**完整的 RAG 流程**：
```
检索（Retrieval）→ 判断（LLM）→ 生成（Generation）
```

## 💡 关键理解与常见误区

### ✅ 关键理解

#### 1. RAG 不是让 LLM 学习新知识

**错误理解**：RAG 把新知识"教给"了 LLM
**正确理解**：RAG 只是在提问时临时提供参考资料

类比：
- ❌ 不是：给学生补课，让他记住新知识
- ✅ 而是：考试时允许翻书查资料

#### 2. 向量检索是"语义搜索"不是"关键词搜索"

**传统搜索**（关键词匹配）：
```
问题："图书馆开门时间"
文档："图书馆开放时间是..."
结果：❌ 找不到（因为"开门"≠"开放"）
```

**向量检索**（语义匹配）：
```
问题："图书馆开门时间"
文档："图书馆开放时间是..."
结果：✅ 能找到（理解"开门"="开放"）
```

#### 3. chunk_size 需要根据实际情况调整

**不是越大越好，也不是越小越好**：

| chunk_size | 优点 | 缺点 | 适用场景 |
|-----------|------|------|---------|
| 小（200-300） | 检索精准 | 可能切断上下文 | 问答型文档 |
| 中（500-800） | 平衡 | 通用 | 大多数场景 |
| 大（1000+） | 上下文完整 | 检索不精准，成本高 | 长文档分析 |

**本项目使用 500**：适合校园问答场景（每个问题答案较短）

#### 4. LangChain 是工具箱，不是完整应用

**LangChain 提供**：
- ✅ 组件（LLM、Retriever、Prompt）
- ✅ 设计模式（LCEL 链式语法）
- ✅ 集成（向量库、文档加载器）

**你仍需要做**：
- ❌ 设计业务逻辑
- ❌ 编写 API 接口
- ❌ 设计 Prompt
- ❌ 处理错误和优化

#### 5. 向量化只需要做一次

**构建阶段**（离线）：
- 文档 → 向量 → 保存到磁盘
- 调用 OpenAI API，有成本

**查询阶段**（在线）：
- 只需要把问题向量化（1次 API 调用）
- 从磁盘加载已有索引（免费）

💰 **成本优化**：7个文档块的向量化成本 < $0.0001

### ❌ 常见误区

#### 误区1：文档越多越好

**错误想法**：把所有文档都塞进知识库，越多越好。

**实际问题**：
- 检索时间变长
- 可能检索到不相关内容
- 噪音增加，影响准确率

**正确做法**：
- 只添加高质量、相关的文档
- 定期清理过时内容
- 对文档进行分类和标签

#### 误区2：检索到的就是正确答案

**错误想法**：FAISS 检索到的文档一定能回答问题。

**实际情况**：
- 可能检索到相关但不准确的内容
- 知识库中可能没有答案
- 需要 LLM 判断和综合

**正确做法**：
- 让 LLM 判断检索结果是否相关
- 设计 Prompt 处理"无答案"情况
- 添加置信度评估

#### 误区3：LangChain 是完整应用框架

**错误想法**：安装 LangChain 就能直接运行问答系统。

**实际情况**：
- LangChain 只是组件库
- 需要自己设计架构
- 需要编写业务代码

**类比**：
- LangChain ≈ React（组件库）
- 不是 Django（完整框架）

#### 误区4：向量相似度 = 答案质量

**错误想法**：相似度 0.9 的文档一定比 0.7 的好。

**实际情况**：
- 相似度只表示语义接近
- 不代表内容质量
- 可能检索到相似但错误的信息

**正确做法**：
- 结合多个文档块
- 让 LLM 综合判断
- 添加来源验证
